Events

An Event is a trigger that causes all or part of your code to run.

Prior to TradeStation 9, only two events existed.
     1. A new tick arriving from the market.
     2. End of bar.

These two events caused (and still cause) your code to be processed from top to bottom (with exceptions, see Understanding how EasyLanguage Code is Processed).

In TradeStation 9 a whole new set of Events were introduced. Now an Event can be anything from '1 second has passed' to 'my order was filled' to 'the fifth level of the market depth window has changed'. With these new Events, rather than your whole code being processed, only a specific section of EasyLanguage code, called a Method, is run.

As an example, this piece of EasyLanguage code declares a timer and sets a Method to be processed whenever the Timer Event is triggered.

using elsystem;

var: Timer Timer1( NULL );


method void Timer1_Elapsed( Object sender, TimerElapsedEventArgs args ) 
begin
	//Do something on the event
end;


once 
begin
	Timer1 = new elsystem.Timer;
		Timer1.Interval = 1000;
		Timer1.AutoReset = true;
		Timer1.Enable = true;
		Timer1.Name = "Timer1";
	
	Timer1.elapsed += Timer1_Elapsed;
end;

In the above code the Timer is declared as Timer1. There is an event called elapsed which is part of the Timer Class. So the line

     Timer1.elaped += Timer1_Elapsed

says that if the elapsed Event is triggered then run the Timer1_Elapsed method.

Earlier in the code the line

     method void Timer1_Elapsed( Object sender, TimerElapsedEventArgs args )

declares the method which will be called whenever the elapsed Event is triggered.

Note the parameters used when calling an Event method. In the above example we have 'Object sender' and 'TimerElapsedEventsArgs args'. These are used to pass system information about the Event to the method. Sometimes these parameters are useful, and other times not, but they are always required parameters. Don't get hung up on these parameters. Just be aware that they must be there, and if you don't know what they are just copy and paste them from the documentation.


Programming Notes:

  1. The terminology for talking about Events is not exact. A new tick arriving from the market or an end of bar signal may be termed an event (with a small e). These cause your code to be run from top to bottom. An Event (with a capital E, introduced in TradeStation 9) is caused by something else (i.e. not a new tick or end of bar) and only processes a specific section of EasyLanguage code.

  2. Events should be setup only after the Object is fully declared. In the above example, we create the Timer, set its properties, and then declare the Event. It you declare the Event before the properties, then it's possible that the Event will fire before the object is fully setup, resulting in an error.

  3. Events (with a capital E) do not happen during normal code processing. If an order is filled, for example, then your normal code will run until the end and then the method associated with the Event will be run. Events do not interrupt normal code processing, nor do Events interrupt methods or other Events.

  4. There are at least two consequences of number 3. First, the BarStatus reserved word will always return a -1 value if used within an Event. This is because you are not processing intrabar or end-of-bar, but 'out of time'. Second, any variables that are modified by Events must be declared as IntrabarPersist.
